# 技術設計 — obsidian-jsx-renderer

## アーキテクチャ概要

JSXソースコードをランタイムでReactコンポーネントに変換し、Obsidian上でインタラクティブにレンダリングするプラグイン。

```
JSXソース → Sucrase(トランスパイル) → new Function(評価) → createRoot(レンダリング)
```

## モジュール構成

| ファイル | 責務 |
|---|---|
| `src/main.ts` | プラグインエントリポイント。ビュー・コードブロックプロセッサ・設定タブの登録 |
| `src/jsx-renderer.ts` | JSX→React変換・レンダリングのコアエンジン |
| `src/jsx-codeblock.ts` | マークダウン内 ` ```jsx ``` ` コードブロックの処理 |
| `src/jsx-view.ts` | `.jsx` ファイルを開いた時のカスタムビュー（`TextFileView` 継承） |
| `src/settings.ts` | プラグイン設定の定義と設定画面 |
| `styles.css` | UIスタイル（Obsidian CSS変数を使用してテーマ対応） |

## レンダリングパイプライン

### 1. トランスパイル（Sucrase）

```typescript
transform(jsxSource, {
  transforms: ["jsx", "imports"],
  jsxRuntime: "classic",
  production: true,
});
```

- **Sucrase** を採用。Babelと比較して軽量（~100KB）かつ高速
- `jsx` transform: JSX構文を `React.createElement()` に変換
- `imports` transform: ESM `import/export` を CJS `require/module.exports` に変換

### 2. モジュール解決（mockRequire）

トランスパイル後のCJSコードが `require()` を呼ぶ際、プラグインにバンドルされたReactを返す。

```typescript
const mockRequire = (id: string) => {
  if (id === "react" || id === "React") return React;
  if (id === "react-dom/client") return ReactDOM;
  if (id === "react-dom") return ReactDOM;
  throw new Error(`未対応のモジュール: ${id}`);
};
```

対応モジュールはReact/ReactDOMのみ。外部依存のないコンポーネントが対象。

### 3. コード評価（new Function）

```typescript
const fn = new Function("require", "module", "exports", "React", code);
fn(mockRequire, moduleObj, moduleObj.exports, React);
```

`new Function()` でトランスパイル済みコードを評価し、`module.exports.default` からコンポーネントを取得。

### 4. レンダリング（React 18 createRoot）

```typescript
const root = ReactDOM.createRoot(wrapper);
root.render(React.createElement(ErrorBoundary, {
  onError: (error) => showError(error),
  children: React.createElement(Component),
}));
```

`ErrorBoundary`（クラスコンポーネント）でランタイムエラーをキャッチし、UIにエラーメッセージを表示。

## 2つのレンダリングモード

### コードブロックモード（jsx-codeblock.ts）

- マークダウン内の ` ```jsx ``` ` を `registerMarkdownCodeBlockProcessor` でフック
- ツールバー付きコンテナを生成し、ソース/レンダリングの切り替えボタンを提供
- `MutationObserver` でDOMからの除去を検知し、Reactルートをアンマウント

### ファイルビューモード（jsx-view.ts）

- `.jsx` 拡張子を `registerExtensions` で `JsxFileView` に関連付け
- `TextFileView` を継承し、ファイル内容の変更を検知して再レンダリング
- ツールバーにソース切替 + 手動再レンダリングボタン
- `autoRender` 設定がオフの場合はプレースホルダーを表示し、明示的なボタン押下でレンダリング

## エラーハンドリング

2段構えのエラーキャッチ:

1. **トランスパイル/評価時エラー**: `try/catch` でキャッチ（構文エラー、未対応モジュール等）
2. **ランタイムエラー**: `ErrorBoundary`（React `getDerivedStateFromError` + `componentDidCatch`）でキャッチ

いずれもUIにエラータイトルと詳細メッセージを表示。`showErrorDetails` 設定で詳細の表示/非表示を制御。

## 設定項目

| 設定 | デフォルト | 説明 |
|---|---|---|
| `defaultShowSource` | `false` | 初期表示をソースコードにするか |
| `showErrorDetails` | `true` | エラー時に詳細メッセージを表示するか |
| `autoRender` | `true` | `.jsx` ファイルを開いた時に自動レンダリングするか |

## セキュリティ上の考慮

- `new Function()` による動的コード実行を使用しており、任意のJavaScriptが実行可能
- 実行されるのはVault内のコード（ユーザー自身が配置したファイル）のみ
- 個人利用を想定したプラグインであり、信頼できるソースからのJSXのみを扱う前提

## 技術スタック

- **TypeScript** — 型安全なプラグイン開発
- **React 18 + ReactDOM** — コンポーネントランタイム（プラグインにバンドル）
- **Sucrase** — 軽量JSXトランスパイラ（ランタイム使用）
- **esbuild** — バンドラ（ビルド時のみ）
- **Obsidian Plugin API** — ビュー登録、コードブロック処理、設定画面

## 対象JSXの前提

- `export default function Component()` 形式のdefault export
- React Hooks（`useState`, `useEffect` 等）使用可
- インラインスタイルによるスタイリング
- 外部依存なし（React本体のみ使用可能）

## 不採用とした代替案

| 選択肢 | 不採用理由 |
|---|---|
| Babel | バンドルサイズが大きすぎる（数MB）。Sucraseは~100KBで十分 |
| obsidian-react-components（既存プラグイン） | 2022年以降メンテナンス放棄。CM6対応が壊れており最新Obsidianで動作しない |
| iframe サンドボックス | Obsidian APIとの連携が困難。個人利用では過剰な隔離 |
